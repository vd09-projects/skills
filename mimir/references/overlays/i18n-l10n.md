# Internationalization & Localization Overlay

Composable concern: change touches user-facing strings, locale-aware formatting (dates, numbers, currency, units), pluralization, gender, RTL layout, character encoding, translation pipeline. Forces explicit locale planning so the change doesn't silently break a locale you don't speak.

Applies to both frontend (UI strings, layout direction, input methods) and backend (locale-aware emails, PDFs, exports, API responses with locale-dependent content).

## Triggers

Activate when one or more present:

- Keywords: `i18n`, `l10n`, `locale`, `translation`, `language`, `RTL`, `right-to-left`, `LTR`, `Unicode`, `UTF-8`, `pluralization`, `ICU`, `MessageFormat`, `gettext`, `currency`, `timezone`, `date format`, `number format`
- Library signals: imports from i18n libs (i18next, react-intl / FormatJS, vue-i18n, ngx-translate, polyglot, Lingui, react-i18next, Linguist)
- Path/file signals: `locales/`, `translations/`, `i18n/`, `messages/`, `.po`, `.pot`, `.xliff`, `.json` files with locale codes
- Phrases: "support {language}", "ship to {region}", "RTL languages", "Japanese / Arabic / Hebrew / German", "format the price", "format the date"

Do not activate for: changes scoped to user-facing strings in only the default development locale AND the project has no other locales today AND none planned, internal-only admin tools with documented locale exemption.

## Required Slots

1. **Locale matrix.** Which locales are in scope — language code + region (e.g., `en-US`, `pt-BR`, `zh-Hans-CN`, `ar-EG`). Plus declared fallback chain.
2. **Translation source-of-truth.** Where translation strings live — files in repo, TMS (Lokalise, Phrase, Crowdin, Transifex), database. Drives the workflow for new strings.
3. **Translation pipeline.** How strings flow from author → translator → production. Manual extraction? CI extraction? Pre-translation by MT? Review process?
4. **Pluralization & gender rules.** Any new strings with counts, gender, or ordinal references — must use a plural/select format, not concatenation.
5. **Locale-aware formatting.** Dates, numbers, currencies, units, names, addresses — formatted via the platform's locale APIs (Intl in JS, `strftime` is NOT a locale API).
6. **RTL handling.** If any in-scope locale is RTL (Arabic, Hebrew, Persian, Urdu), the layout, logical CSS properties, and bidi text handling must be planned.
7. **Character / input support.** Encoding (UTF-8 end-to-end), IME support, surrogate pair handling for emoji and CJK extension chars.

### Pluralization & Gender Plan

For each string with counts or gender:

```
{key}:
  one: "{singular form}"
  other: "{plural form}"
  # or full ICU MessageFormat:
  # "{count, plural, one {# item} other {# items}}"
```

- **Languages with non-binary plural categories:** confirm Polish, Russian, Arabic, Welsh handled (one / few / many / other / zero / two).
- **Gender select:** any string that varies by gender uses `{gender, select, ...}`, never string concat.

## Discipline

- **No hardcoded user-facing strings.** Every visible string flows through the i18n layer, including error messages, alt text, ARIA labels, and email subjects.
- **No string concatenation for sentences.** "You have " + count + " items" is a bug in every language with non-trivial plural rules. Use MessageFormat.
- **Locale comes from the user, not the browser default.** Express preference > inferred preference > server default.
- **`new Date().toLocaleString()` with no locale argument is non-deterministic.** Always pass an explicit locale.
- **Translator context is not optional.** "Submit" needs context — submit a form? Submit a job? Submit a tax return? Same English word, different translations.
- **Bidi-aware text means logical CSS, not flipping `left` to `right`.** Logical properties or nothing.
- **A locale shipped at 70% translated is broken in production.** Either gate launches on coverage threshold or document the fallback policy.

## Common Failure Modes

- **"It works in English"** — English is the language that tolerates the most bad i18n. German truncates, Russian pluralizes weirdly, Arabic mirrors, Japanese has no spaces.
- **String concatenation in templates** — `"Hello, " + name + "!"` works in English, breaks in Japanese (greetings vary), Hindi (word order), and any language with honorifics.
- **Hardcoded English in error messages** — "Internal server error" reaches a non-English user with no fallback localization.
- **Date as `MM/DD/YYYY`** — only the US writes dates this way. Almost everywhere else, `DD/MM/YYYY` or ISO.
- **Currency formatted with hardcoded `$`** — `$10` in `es-MX` should be `MX$10` or `10 US$`, depending on context.
- **`.length` to validate username length** — emoji counts as 2+ code units; rejects valid usernames.
- **RTL layout flipped by manually swapping `left` ↔ `right`** — half the bugs come back inverted; logical CSS solves this.
- **Translations stored in the same file as source** — translators edit code, devs break translations.
- **No fallback when a string is missing** — UI shows the raw key (`profile.settings.save`) to the user.
- **Email sent in user's locale, but link in email goes to default locale** — coherence broken across the journey.
