# Decision Journal

Track every significant decision, rejected approach, experiment outcome, and reasoning trail across a project's lifetime. The goal: prevent re-exploring dead ends, preserve the "why" behind choices, and make it possible — months later — to answer "Did we already try X? Why did we choose Y over Z?"

This skill operates in two ways:

- **Standalone:** the user records and queries decisions directly through the modes below.
- **As infrastructure:** other skills mark decisions inline in their responses using the format spec in `references/inline-format.md`, and this skill harvests those marks at end of conversation.

Both modes coexist. Producer skills do not need to know this skill is installed; their inline marks are valuable conversation-level annotations even without harvest. Conversely, this skill does not need to know which producer skills are installed; any inline mark matching the format spec is harvestable.

## What belongs here (and what doesn't)

Record a decision when:
- Someone might ask "why did we do it this way?" in 3 months
- An approach was evaluated and rejected (especially with data or reasoning)
- A tradeoff was consciously made between competing options
- An experiment produced results worth preserving
- A convention or standard was established that affects future work

Do NOT record:
- Routine, obvious, or trivial choices
- Git-level changes (that's what commits are for)
- Session-level context (that belongs to session-continuity skills if present)
- Task/work-item tracking (that belongs to ticket-management skills if present)

## Storage structure

Decisions live in a `decisions/` directory at the project root, organized by category:

```
decisions/
├── INDEX.md              # YAML-based lookup index
├── architecture/
│   └── 2026-03-15-event-driven-over-polling.md
├── library-choice/
│   └── 2026-03-20-chose-redis-over-memcached.md
├── algorithm/
├── convention/
├── infrastructure/
├── performance/
└── tradeoff/
```

The category subdirectories above are defaults. Projects can define custom categories via `.decision-journal.yml` (see Configuration). Create subdirectories on demand — don't pre-create empty ones. **Categories not in the config are accepted, not rejected** — if a producer marks a decision in a new category, create the subdirectory and note it to the user.

File naming: `YYYY-MM-DD-short-slug.md` where the slug is generated by the algorithm in the Slug Generation section below.

## Configuration

If a `.decision-journal.yml` file exists at the project root, read it for overrides. If it doesn't exist, use defaults. Never create this file unless the user asks.

```yaml
# .decision-journal.yml (all fields optional)
decisions_dir: decisions          # default: decisions
categories:                       # default: architecture, library-choice, algorithm, convention, infrastructure, performance, tradeoff
  - architecture
  - library-choice
  - algorithm
  - custom-category
review_threshold_days: 90         # flag decisions older than this for review (default: 90)
default_tags: []                  # tags auto-applied to every decision
```

The `categories` list is a *suggestion*, not a hard constraint. Categories outside the list are accepted on demand.

## Slug generation

When generating a filename slug from a source string (decision title, scope metadata, or first ~80 chars of prose body):

1. Take the source string.
2. Lowercase.
3. Replace anything not in `[a-z0-9]` with a hyphen.
4. Collapse runs of hyphens to a single hyphen.
5. Strip leading and trailing hyphens.
6. If longer than 50 characters, truncate at the last hyphen before character 50.
7. Prepend `YYYY-MM-DD-`.

This algorithm is deterministic — two harvests of the same conversation produce the same filename, which is what the duplicate-detection logic in Harvest mode depends on.

## Decision template

Read the template from `templates/decision.md` in the skill directory. When creating a new decision, only three fields are required: **Title**, **Context**, and **Decision**. Fill in everything you can infer from the conversation, leave the rest blank or omit optional sections entirely.

## INDEX.md format

INDEX.md uses a YAML-based format for machine-friendly querying. Read the template from `templates/INDEX.md` in the skill directory.

Each entry looks like:

```yaml
- id: 2026-03-15-event-driven-over-polling
  title: "Chose event-driven architecture over polling"
  date: 2026-03-15
  status: accepted
  category: architecture
  tags: [events, rabbitmq, polling, scalability]
  path: architecture/2026-03-15-event-driven-over-polling.md
  summary: "Adopted event-driven pattern with RabbitMQ for real-time updates instead of periodic polling."
```

When updating INDEX.md, preserve the existing YAML structure. Add new entries at the top (most recent first). Always include the one-line `summary` field.

## Modes of operation

### 1. Record

**Triggers**: User says "record a decision", "log this decision", "we decided to…", or you detect a significant choice in conversation.

**Proactive nudge behavior**: If you detect a significant decision being made (e.g., "let's go with Redis over Memcached because..."), offer a brief nudge:
> "This looks like a decision worth recording. Want me to log it to the decision journal?"

Only nudge when the choice is non-trivial and involves rejected alternatives or explicit tradeoffs. Don't nudge more than once per conversation unless the user is making clearly distinct decisions. If the user declines, move on — don't ask again for the same decision.

**Steps**:

1. Read `.decision-journal.yml` if it exists to get config overrides.
2. Extract from conversation context: the decision, alternatives considered, reasoning, and any data/metrics mentioned.
3. Determine the category. If ambiguous, ask the user (suggest your best guess).
4. Generate a filename: `YYYY-MM-DD-<slug>.md` using the slug-generation algorithm above and today's date.
5. Create the decision file in `decisions/<category>/`, filling in all fields you can infer. Required: Title, Context, Decision. Leave optional sections blank if not inferrable.
6. Auto-suggest related decisions: scan INDEX.md for entries with overlapping tags or similar titles. If matches are found, present them to the user and ask whether to add cross-links. Only insert Related Decisions links after user approval.
7. Update INDEX.md with the new entry at the top.
8. If this decision supersedes an existing one:
   - Automatically update the old decision's status to `superseded`
   - Add a forward-link in the old decision's Related Decisions section pointing to the new one
   - Update the old entry's status in INDEX.md

### 2. Query

**Triggers**: "Have we tried…?", "Why did we choose…?", "Show me all decisions about…", "What did we reject for…?", any question about past decisions.

**Search strategy**: Start with INDEX.md. Parse the YAML entries and filter by the relevant combination of tags, category, status, title, and summary text. Only open individual decision files if:
- The INDEX.md summary doesn't contain enough detail to answer the question
- The user asks for full context, pros/cons, or experiment data
- Fewer than 3 matches are found (cheap to open them all)

**Response format**: Brief summaries with file paths. For each matching decision, show:
- Title, date, status
- The one-line summary from INDEX.md
- File path for the user to dig deeper

If a decision was rejected, prominently highlight the rejection reason so the user doesn't re-explore a dead end.

### 3. Review

**Triggers**: "What decisions need revisiting?", "Show me stale decisions", "Decision review", or periodically when the user asks for a project health check.

**What to surface** (as an actionable list with concrete next steps):

1. **Revisit-later**: Decisions with status `revisit-later` — show the revisit trigger condition if set.
2. **Stale experimentals**: Decisions with status `experimental` that are older than `review_threshold_days` (default 90) and never got updated to a final status.
3. **Triggered revisits**: Decisions with a `revisit-trigger` field whose conditions may now be relevant (use your judgment based on project context).
4. **Aging decisions**: Accepted decisions older than `review_threshold_days` with revisit triggers — flag if the trigger condition might be met.

**Response format**: An actionable list. For each decision, show the title, age, why it's flagged, and a suggested next action.

### 4. Summarize

**Triggers**: "Summarize all decisions about…", "Decision history for…", "Give me context on…", onboarding-style requests.

**Response format**: A narrative summary — a coherent story of how decisions evolved over time, not just a list. Connect the dots: show how early decisions influenced later ones, where the team changed direction and why, and what the current state of thinking is.

### 5. Harvest

**Triggers**:
- User says "harvest decisions", "extract decisions from this conversation", "capture inline decisions"
- End of conversation, when one or more inline decision marks are present and none have been harvested in this conversation
- After a producer skill makes a series of marks, before the conversation ends

**Proactive nudge behavior**: If inline marks are present in the conversation and the user hasn't triggered harvest, offer once near a natural stopping point:
> "I noticed N inline decision marks in this conversation. Want me to harvest them to the journal?"

Offer once per conversation. If declined, don't ask again.

**What it does**: Extracts inline decision marks from the conversation transcript and converts each to a full decision file in the journal.

**Inline mark format**: Defined in `references/inline-format.md`. The lead-in regex is `\*\*Decision \(\d{4}-\d{2}\.\d+\.\d+\) — [^:]+: [^*]+\*\*` (per the v1 spec; the regex is intentionally permissive within v1 — future versions may need it widened).

**Steps**:

1. Load `references/inline-format.md` for the full parsing algorithm and current format details.

2. Scan the conversation transcript using the lead-in regex. For each match, extract:
   - The lead-in line (capturing version, category, status)
   - The metadata block (key-value pairs after the lead-in until the first blank line)
   - The prose body (everything after the metadata block until the block boundary)
   - See `references/inline-format.md` for boundary detection rules.

3. **Build a confirmation list of all extracted marks before writing anything.** Show the user:
   - The lead-in (category and status)
   - The scope or first 80 chars of the prose body (so the user can identify the decision)
   - The owner if present in metadata

   Format:
   ```
   I found N inline decision marks in this conversation:
   
   1. [convention/experimental] scope: engine/accounting
      "All money values use shopspring/decimal..."
      owner: priya
   
   2. [algorithm/experimental] scope: sizing
      "Half-Kelly with drawdown scaling..."
      owner: marcus
   
   3. [architecture/experimental] (no scope)
      "CPCV will live at internal/cv/cpcv as a sibling..."
   
   Should I write all of these as decision files? Or are any of them duplicates 
   of each other (same decision marked twice) that I should consolidate?
   ```

   Wait for user confirmation before writing.

4. **Cross-conversation duplicate check.** For each mark the user confirmed:
   - Generate the proposed slug using the slug-generation algorithm.
   - Check INDEX.md for entries created within the last 24 hours with the same slug or substantially-similar slug (first 30 chars match).
   - If a match is found, surface to user: *"This decision looks similar to `<existing-slug>` from <date>. Skip this mark, write it as a new decision anyway, or merge with the existing one?"*
   - The 24-hour window catches the case where the user re-runs Harvest on a conversation already harvested. Older potential duplicates are not flagged — those are likely intentional re-decisions or refinements.

5. **Multi-producer mark check.** If two confirmed marks have substantially similar prose bodies but different `owner` metadata values:
   - Surface to user: *"I see two marks that look like the same decision but were made by different producers (`<owner1>` and `<owner2>`). Are these the same decision (I'll keep one and note both producers), or different decisions that happen to look similar (I'll keep both)?"*
   - The user makes the call. Don't auto-merge.

6. **Write each confirmed (and de-duplicated) mark as a decision file:**
   - Determine the category from the lead-in. If the category is not in the project's config, create the category subdirectory and note it: *"Created new category '<X>' — is this intended? You can add it to .decision-journal.yml to make it official."*
   - Generate the filename using the slug algorithm.
   - Create the decision file at `decisions/<category>/<filename>` using the existing template. Fill in:
     - **Title**: from the first sentence of the prose body, or from the scope metadata if more specific.
     - **Date**: today.
     - **Status**: from the lead-in.
     - **Category**: from the lead-in.
     - **Tags**: from the `tags` metadata key if present.
     - **Context**: from the conversation context surrounding the mark (a few sentences before and after, paraphrased).
     - **Decision**: from the prose body of the mark.
     - **Consequences, Related decisions, Experiments, Revisit trigger**: from metadata keys if present (`related`, `revisit-trigger`), otherwise leave blank.
   - If the mark has a `supersedes` metadata key, update the superseded decision's status to `superseded` and add a forward-link.
   - Update INDEX.md with the new entry.

7. **Mixed-version marks.** If a mark has a version string the current spec doesn't recognize, store it raw with a `version-not-understood` flag in the decision file's metadata, rather than dropping it. The format spec's forward-compatibility rule applies.

8. **No marks found.** If the conversation contains no inline marks, say so plainly: *"No inline decision marks found in this conversation."* Do not offer to record decisions manually — that's Record mode and the user can trigger it explicitly if they want.

9. **Summary report.** After all writes, summarize what was done:
   ```
   Harvested 3 decisions:
   - decisions/convention/2026-04-15-money-as-decimal.md (new)
   - decisions/algorithm/2026-04-15-half-kelly-drawdown-scaling.md (new)
   - decisions/architecture/2026-04-15-cpcv-package-location.md (new)
   
   Updated INDEX.md.
   Skipped 1 mark as duplicate of existing decision.
   ```

## Cross-skill integration

### Inline decision marking (producers)

Other skills may mark decisions inline in their responses using the format specified in `references/inline-format.md`. These marks are designed to be harvested by this skill's Harvest mode at end of conversation.

This skill does not need to know which producer skills are installed. Any inline mark matching the format spec is harvestable, regardless of which skill produced it. Producers commonly include:

- Skills that make architectural or convention decisions (assign `convention`, `architecture`, or `tradeoff` categories).
- Skills that make algorithm or methodology decisions (assign `algorithm` category).
- Skills that flag tradeoffs or override default rules (assign `tradeoff` category).

The category assignment is producer-defined; this skill accepts any category and creates subdirectories on demand.

### Producer-agnostic operation

The journal does not invoke, configure, or depend on any producer skill. If producer skills are present and emit marks, the journal harvests them. If no producer skills are present, the journal operates standalone via Record, Query, Review, and Summarize.

If a producer skill is *not* installed but its marks remain in conversation history (e.g., from a prior conversation), the journal can still harvest them — the format is producer-agnostic.

### Inline marks without a journal

Inline decision marks are valuable annotations even when this skill is not installed. They are readable inline in conversation; they capture reasoning at the moment of decision; they document the why for human readers. The journal makes them durable across conversations and queryable; without the journal, they remain useful but ephemeral.

## Maintenance notes

- When editing any decision file, always keep INDEX.md in sync.
- When a decision is superseded, update both the old file and INDEX.md in the same operation.
- If a category subdirectory doesn't exist yet, create it when writing the first decision in that category.
- Never delete decision files. Rejected and superseded decisions are valuable history.
- Keep INDEX.md entries in reverse chronological order (newest first).
- Harvest mode does not delete or modify the inline marks in the source conversation — it only reads them and creates files. Re-running Harvest on the same conversation will trigger the cross-conversation duplicate check.
