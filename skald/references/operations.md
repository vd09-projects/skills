# Skald — Operation Details

Authority for the persistence steps summarized in SKILL.md. SKILL.md describes the flow; this file is the source of truth for edge cases.

---

## Slug Resolution — full algorithm

Inputs:
- User request (free text)
- Optional `--scope <slug>` arg
- Target skill's output (only available AFTER skill runs; for pre-run slug resolution, scope arg + scope_hint of user request only)

Order of operations:

### Pre-run resolution (before invoking target skill)

If `--scope` was passed:

1. Validate slug format: kebab-case, lowercase, ≤40 chars, `[a-z0-9-]` only.
2. Check existence in `scopes.md` or `.claude/handoff/`. If yes → use. If no → confirm with user: "Scope `{slug}` does not yet exist. Create new with this slug? [y/n/supply-alternative]".
3. Proceed to invoke target skill, passing the scope dir path.

If `--scope` was NOT passed:

1. Derive candidate from the user's free-form request. Pick the noun phrase that names the work. If ambiguous, ask user: "What's the scope of this work? Give me one phrase."
2. Apply slug-style transformation per `config.md` (kebab, kebab-noun, prefixed).
3. Match against `scopes.md`. Match rules:
   - **Exact** (candidate equals known slug or alias) → use existing, no prompt.
   - **Close** (string similarity ≥ `confirm_existing_match_threshold`, default 0.7) → prompt with `[r] reuse | [n] create new | [a] add as alias | [c] cancel`.
   - **None** → prompt with `[y] confirm | [s] supply alternative | [c] cancel`.
3. On confirmation of a new scope, capture reasoning from user before proceeding. Reasoning required field.

### Post-run reconciliation

After target skill runs, its output frontmatter contains `scope_hint`. Compute candidate slug from `scope_hint` and compare with the pre-run resolved slug:

- **Same slug** → no action.
- **Different slug** → prompt user: "Skill output suggests different scope: `{output-slug}`. Pre-resolved: `{prerun-slug}`. Reconcile: [k] keep prerun (write here) | [m] move to output's slug (create new scope or append to existing) | [a] add output-slug as alias of prerun | [c] cancel persistence".

This reconciliation step exists because user-supplied request phrasing can be vague; skill output (which has the full picture after thinking) may suggest a better scope.

## Iteration / version computation

Given a write target `.claude/handoff/{slug}/{canonical}.md`:

```
exists = file_exists(target)

if exists:
    existing_frontmatter = read_frontmatter(target)
    current_version = existing_frontmatter.get("version", 1)
    new_version = current_version + 1
    archive_path = ".claude/handoff/{slug}/_history/{canonical}-v{current_version}.md"
    move(target, archive_path)
    prior_versions = list_history_files(".claude/handoff/{slug}/_history/", canonical_stem=canonical)
else:
    new_version = 1
    prior_versions = []
```

`list_history_files` returns filenames sorted by version number ascending. The new artifact's `prior_versions` field holds this list.

`created` is preserved from the existing file's frontmatter when iterating; only `updated` advances. For first version, `created = updated = now`.

## Index update — INDEX.md

File location: `.claude/handoff/INDEX.md`. Format (default `markdown`):

```markdown
# Handoff Index

Generated by Skald. Do not edit manually — rerun Skald or wait for next write.

| Slug | Title | Status | Last Skill | Version | Updated | Owner |
|------|-------|--------|------------|---------|---------|-------|
```

Update algorithm:

1. Read existing INDEX.md, parse rows by slug.
2. Find row for current slug.
3. If exists: update Status, Last Skill, Version, Updated columns. Owner column unchanged unless user provided new value.
4. If absent: insert new row.
5. Sort all rows by `Updated` descending.
6. Rewrite INDEX.md preserving header.

If `config.md` sets `index_format: yaml`, replace the table with a YAML list:

```yaml
scopes:
  - slug: auth-redesign
    title: "JWT rotation + session invalidation"
    status: review-pending
    last_skill: review-findings
    version: v1
    updated: 2026-05-21
    owner: vd
```

## Log update — LOG.md

File location: `.claude/handoff/LOG.md`. Append-only.

```markdown
# Handoff Log

Append-only. Skald appends one row per artifact write.

| Date | Scope | Role | Plan Type | Version | Status | Notes |
|------|-------|------|-----------|---------|--------|-------|
| 2026-05-20 14:30 | auth-redesign | planner | architecture | v1 | draft | initial ADR |
```

Append at end. Never rewrite. Never remove. Date is `YYYY-MM-DD HH:MM` local time (or UTC if `config.md` sets so — TODO future field).

`Notes` column is a one-line summary, taken from the artifact body's first paragraph after the title, OR a `summary:` field in the frontmatter if the producer supplied one. Truncate to 80 chars.

## Thread update — `_thread.md`

File location: `.claude/handoff/{slug}/_thread.md`. Auto-created if absent.

First-time creation includes a header:

```markdown
# Thread — {slug}

{Scope title from scopes.md or the producer's scope_hint}

---
```

Then one entry per write:

```markdown
## {YYYY-MM-DD HH:MM} — {producer_role} {plan_type} v{N} ({status})

{Summary — one paragraph, from artifact body or `summary:` field}
```

Append-only. Never rewrite. Status in the heading captures the artifact's status at write time (always `draft` when skald writes); user-driven status changes are not auto-mirrored into `_thread.md` in v2 MVP.

## Status aggregation — full table

Used to compute `INDEX.md` Status column. Apply against the scope's latest-updated artifact:

| Artifact role + status | INDEX.md Status |
|---|---|
| no artifacts in scope dir | `new` |
| `planner-architecture` status `draft` | `arch-pending` |
| `planner-architecture` status `approved`, no task yet | `arch-approved` |
| `planner-architecture` status `consumed` | `arch-consumed` |
| `planner-task` status `draft` | `task-plan-draft` |
| `planner-task` status `approved` | `ready-to-build` |
| `planner-task` status `consumed` | `building` |
| `implementation-build` status `draft` | `build-in-progress` |
| `implementation-build` status `approved` | `ready-for-review` |
| `implementation-build` status `consumed` | `review-consumed` |
| `review-findings` status `draft` | `review-pending` |
| `review-findings` status `approved` AND findings ≥ 1 blocker | `changes-requested` |
| `review-findings` status `approved` AND findings = 0 blockers | `approved-clean` |
| `review-findings` status `consumed` | `iterated` |

Tie-breaking when two artifacts updated at same minute: review > implementation > planner ordering wins for Status display.

Override via `config.md`'s `status_overrides` map.

## Validation — producer output checks

Before persisting, validate the producer's emitted frontmatter:

| Check | Failure mode |
|---|---|
| `artifact_type: handoff` present | Reject. Output is not a handoff artifact. |
| `artifact_version: 2` present | Reject. Producer is on wrong protocol version. |
| `producer_role` in role catalog | Reject. Unknown role. List acceptable values. |
| `plan_type` present | Reject. Required for canonical filename. |
| `canonical_name` matches `{producer_role}-{plan_type}` | Warn + override. Producer suggested wrong name; skald uses canonical computation. |
| `scope_hint` present | Reject. Required for post-run reconciliation. |
| `status` is `draft` | Reject if producer wrote `approved` or `consumed`. Producer cannot self-approve. |

On any rejection: do NOT persist. Report what's missing. The producer's output is held in memory (in conversation) until either fixed or abandoned.

## Edge cases

### Two skills declare same role

Allowed but discouraged. If two skills both declare `producer_role: planner` and run on the same scope, the second run archives the first to `_history/`. Both share the canonical `planner-architecture.md` namespace. To avoid clobbering, configure one of them in `never_overlays` or run on different scopes.

### Skill produces multi-part output

A skill might want to emit BOTH an architecture AND a task plan in one invocation. Treat as two separate persists: skald validates and writes each `plan_type` separately. The producer must emit two complete frontmatter+body blocks back to back, clearly delimited.

### Skill output has no body content

Reject. An empty artifact is not useful. Producer must either supply body or return its own `Blocked` terminal state (in which case skald does not invoke persistence).

### Multiple writes in rapid succession (same skill, same scope)

Each write produces a new version. v1 → archive → v2 → archive → v3. No collapse. The thread + log show every write.

### Outer agent invokes skald in dry-run mode

Future v2.1: `--dry-run` flag would invoke target skill but skip the write step. Useful for "what would skald write?" preview. Not in v2 MVP.
